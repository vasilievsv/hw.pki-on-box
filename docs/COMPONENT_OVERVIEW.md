# Component Overview: анатомия PKI-on-Box

## Предисловие: шесть слоёв и одна идея

Если разрезать PKI-on-Box сверху вниз, обнаружится шесть слоёв. Не три (frontend-backend-database), не два (client-server) — именно шесть. И каждый существует потому, что без него система была бы либо небезопасной, либо неработоспособной.

```
+-------------------------------------------------------------+
|                      API Layer                              |
|  REST API (Flask :5000)          CLI                        |
+-------------------------------------------------------------+
|                    Services Layer                           |
|  CA Service    Cert Service    CRL Service    OCSP Responder|
+-------------------------------------------------------------+
|                 Core Layer (Crypto Chain)                   |
|  Self Tests    TRNG    NIST DRBG    CryptoEngine  KeyStorage|
+-------------------------------------------------------------+
|                    Storage Layer                            |
|  PKIDatabase (SQLite)            FileStorage (PEM/DER)      |
+-------------------------------------------------------------+
|                 Security Layer (BSW)                        |
|  SELinux (2 domains)   eBPF (network+syscall)   systemd     |
+-------------------------------------------------------------+
|                   Firmware Layer                            |
|  STM32 TRNG HID (G474/G431/H750, bare metal, USB HID)       |
+-------------------------------------------------------------+
```

Идея, которая объединяет все шесть: каждый слой доверяет только тому, что ниже него. API доверяет Services. Services доверяют Core. Core доверяет Firmware. Security не доверяет никому — он работает на уровне ядра и ограничивает всех остальных. Это не микросервисная архитектура и не гексагональная — это цепочка доверия, реализованная в коде.

## API Layer: два языка одной системы

У PKI-on-Box два интерфейса, и они говорят на разных языках — не потому что так удобнее, а потому что обслуживают разные модели угроз.

### REST API — `api/rest_api.py`

Flask на порту 5000, 12 endpoints. Это язык машин: JSON на входе, JSON (или PEM) на выходе. Никакого HTML, никаких форм, никакого JavaScript. REST API существует для одной цели — автоматизация. Скрипт provisioning, который выпускает 100 сертификатов для IoT-устройств, не должен парсить CLI output.

12 endpoints — не много и не мало. Каждый соответствует одной бизнес-операции:

| Endpoint | Method | Что делает | Почему отдельный |
|----------|--------|-----------|-----------------|
| /api/v1/health | GET | Проверка здоровья | Единственный endpoint без бизнес-логики — для мониторинга |
| /api/v1/ca/root | POST | Создать Root CA | Критическая операция, отдельный audit trail |
| /api/v1/ca/intermediate | POST | Создать Intermediate CA | Менее критичная, но всё ещё CA |
| /api/v1/ca | GET | Список CA | Read-only, безопасно кэшировать |
| /api/v1/ca/{id}/cert | GET | Получить сертификат CA | Публичная информация, можно отдавать без аутентификации |
| /api/v1/certs/server | POST | Выпустить серверный сертификат | RSA-2048, SAN обязателен |
| /api/v1/certs/client | POST | Выпустить клиентский сертификат | EC P-384, другой KeyUsage |
| /api/v1/certs/firmware | POST | Выпустить firmware сертификат | RSA-2048, 5 лет, CODE_SIGNING |
| /api/v1/certs | GET | Список сертификатов | Read-only, пагинация |
| /api/v1/crl/revoke | POST | Отозвать сертификат | Деструктивная операция |
| /api/v1/crl/{ca_id} | GET | Получить CRL | PEM, кэшируется на 24 часа |
| /api/v1/ocsp/{serial} | GET | OCSP статус | Кэшируется на 1 час |

Почему три отдельных endpoint для сертификатов (`/certs/server`, `/certs/client`, `/certs/firmware`), а не один `/certs` с параметром `type`? Потому что у каждого типа — разные обязательные поля, разные алгоритмы, разные сроки жизни. Один endpoint с десятком optional-полей — это приглашение к ошибке: забыл `san_dns` для серверного сертификата — получил сертификат, который Chrome отвергнет. Три явных endpoint — три контракта, каждый валидируется независимо.

Аутентификации на уровне приложения нет. Это не упущение — это решение. Аутентификация реализована ниже: SELinux ограничивает сетевой доступ, eBPF фильтрует по IP. Добавлять API-ключи поверх этого — это defense in depth, но и дополнительный код, дополнительные баги, дополнительная поверхность атаки. На embedded-устройстве, где каждая строка кода — потенциальная уязвимость, минимализм — это безопасность.

### CLI — `api/cli.py`

CLI, entrypoint через `pki.py`. Это язык людей: команды, флаги, human-readable output. CLI существует для операций, которые нельзя доверить сети: создание CA, церемонии, экспорт ключей.

| Command | Subcommand | Зачем отдельно от REST |
|---------|-----------|----------------------|
| ca | create-root | Церемония — только локально, только человек |
| ca | create-intermediate | Создание CA — редкая операция, требует внимания |
| ca | list | Удобство администратора |
| cert | issue-server | Может быть и через REST, но CLI удобнее для одиночных |
| cert | issue-client | То же |
| cert | issue-firmware | То же |
| cert | list | Диагностика |
| crl | revoke | Деструктивная операция — лучше через CLI с подтверждением |
| crl | generate | Ручная генерация CRL |
| crl | check | OCSP проверка из терминала |

CLI и REST API используют одни и те же Services — разница только в точке входа. Это важно: нет двух реализаций бизнес-логики, нет рассинхронизации. `cli.py` вызывает `CertificateAuthorityService.create_root_ca()` — тот же метод, что и REST handler.

## Services Layer: четыре фазы жизни сертификата

Сертификат рождается, живёт и умирает. Четыре сервиса — четыре фазы этого цикла. Они живут в одном процессе (не микросервисы), но разделены по ответственности так, что каждый можно понять, протестировать и заменить независимо.

### CA Service — `services/ca_service.py` — фундамент

Без CA нет PKI. `CertificateAuthorityService` — это класс, который управляет самым ценным активом системы: приватными ключами CA.

Два метода создания: `create_root_ca()` и `create_intermediate_ca()`. Разница — не только в параметрах, а в философии. Root CA — self-signed, живёт 20 лет, `pathlen=None` (неограниченная глубина цепочки). Он создаётся один раз при церемонии и больше никогда не используется для подписи конечных сертификатов. Его ключ расшифровывается только при создании Intermediate CA — раз в 10 лет.

Intermediate CA — подписан Root, живёт 10 лет, `pathlen=0` (не может создавать под-CA). Он подписывает все конечные сертификаты. Если скомпрометирован — отзываем и создаём новый, Root не трогаем. Это стандартная двухуровневая иерархия, и она существует ради одного сценария: компрометация рабочего CA не убивает всю PKI.

In-memory cache `_certs` заполняется при инициализации из SQLite. Каждый `get_ca_cert()` — обращение к dict, не к БД. На RK3328 это разница между 0.001ms и 5ms. При выпуске 100 сертификатов — между 0.1ms и 500ms. Кэш не инвалидируется в runtime — новый CA появляется только при явном создании, и тогда он добавляется в кэш сразу.

`sign_csr()` — метод для подписи Certificate Signing Request. CSR приходит от клиента, содержит публичный ключ и запрошенные extensions. CA копирует extensions из CSR, добавляет issuer_name и подписывает. Validity — 2 года по умолчанию. Это generic-метод; специализированные `issue_*` в CertificateService используют его внутренне, но он доступен и напрямую — для нестандартных сценариев.

### Certificate Service — `services/certificate_service.py` — конвейер

Если CA Service — это фундамент, то Certificate Service — это конвейер. Три метода, три типа сертификатов, три модели использования. Но внутри — один паттерн.

`_base_builder()` — приватный метод, который создаёт заготовку сертификата: subject (CN), issuer (из CA cert), serial (случайный), validity (от now до now + N дней), BasicConstraints (ca=False). Это общее для всех типов. Каждый `issue_*` метод берёт эту заготовку и добавляет свои extensions.

Почему Template Method без наследования? Потому что в Python наследование для трёх вариантов — overengineering. Три метода в одном классе, общий `_base_builder()` — проще, читаемее, тестируемее. Если завтра появится четвёртый тип сертификата (например, email/S-MIME) — это один новый метод, не новый класс.

Три типа — три разных мира:

Server (RSA-2048, 1 год): TLS — самый массовый сценарий. RSA-2048 потому что совместимость важнее длины ключа — есть клиенты, которые не поддерживают EC. SAN обязателен — без него Chrome 58+ отвергнет сертификат. `keyEncipherment` в KeyUsage — для RSA key exchange в TLS (не нужен для ECDHE, но нужен для legacy cipher suites).

Client (EC P-384, 1 год): аутентификация пользователей. EC потому что клиентские сертификаты часто живут на смарт-картах, где размер имеет значение. P-384 (не P-256) — потому что NIST рекомендует P-384 для security strength 192 бит, что соответствует RSA-7680. Запас на будущее. `contentCommitment` — non-repudiation: подпись этим ключом юридически значима.

Firmware (RSA-2048, 5 лет): подпись прошивок. CODE_SIGNING в ExtendedKeyUsage. 5 лет — потому что устройство в поле может работать годами без связи с CA. RSA (не EC) — потому что верификация RSA подписи на embedded-устройстве (Cortex-M0/M3) быстрее, чем EC.

### CRL Service — `services/crl_service.py` — механизм отзыва

Отзыв — это единственная деструктивная операция в PKI. Создание сертификата — добавление. Отзыв — вычёркивание. И вычеркнуть нужно так, чтобы все об этом узнали.

`revoke_certificate()` — два SQL-запроса: INSERT в `revoked_certificates` и UPDATE в `certificates`. Два запроса, не один, потому что это две разные проекции одного факта. CRL генерируется из `revoked_certificates` (нужны только serial + reason). Поиск сертификатов использует `certificates.status` (нужен быстрый фильтр).

`generate_crl()` — тяжёлая операция. Загрузить CA ключ (расшифровать .enc), получить все отозванные serial для этого CA, построить CRL, подписать SHA-256. `next_update = now + 1 день` — обещание актуальности. Клиент может кэшировать CRL на сутки.

Почему CRL, а не только OCSP? Потому что CRL работает офлайн. Устройство в поле скачивает CRL при синхронизации и проверяет сертификаты локально, без сети. OCSP требует онлайн-доступа к PKI — не всегда возможно.

### OCSP Responder — `services/ocsp_service.py` — онлайн-проверка

OCSP — это «спроси и получи ответ». Клиент отправляет serial number, получает GOOD/REVOKED/UNKNOWN. Быстро, точечно, но требует сети.

`OCSPResponder` не имеет собственного хранилища. Он делегирует `is_revoked()` в CRLService, который обращается к SQLite. Единый источник истины — одна таблица `revoked_certificates`. Это исключает ситуацию, когда CRL говорит «отозван», а OCSP — «валиден».

Три статуса — три уровня уверенности. GOOD — «я знаю этот сертификат, он не отозван». REVOKED — «я знаю этот сертификат, он отозван, вот причина и время». UNKNOWN — «я не знаю этот сертификат». UNKNOWN — не ошибка. Это честный ответ, который говорит клиенту: «спроси у другого CA».

## Core Layer: пять звеньев криптографической цепочки

Core Layer — это место, где математика встречается с физикой. Пять компонентов, выстроенных в строгую последовательность: каждый следующий зависит от предыдущего, и эту цепочку нельзя разорвать, переставить или обойти.

### Self Tests (KAT) — `core/self_tests.py` — проверка перед стартом

Прежде чем система начнёт генерировать ключи, она должна убедиться, что умеет это делать правильно. Шесть Known Answer Tests — шесть вопросов, на которые криптографическая библиотека должна ответить правильно:

| Тест | Алгоритм | Эталон | Что проверяет |
|------|----------|--------|--------------|
| _kat_aes_gcm | AES-256-GCM | NIST test vector | Симметричное шифрование работает |
| _kat_hmac_sha256 | HMAC-SHA256 | RFC 4231 "Jefe" | MAC работает |
| _kat_sha256 | SHA-256 | "abc" | Хеширование работает |
| _kat_hmac_drbg | HMAC_DRBG | Zero seed | DRBG инициализируется корректно |
| _kat_rsa_sign | RSA-2048 PKCS1v15 | Runtime keygen | Асимметричная подпись RSA работает |
| _kat_ecdsa_sign | ECDSA P-384 | Runtime keygen | Асимметричная подпись EC работает |

Первые четыре теста — детерминированные: известный вход, известный выход. Если AES-256-GCM с NIST test vector даёт неправильный ciphertext — OpenSSL повреждён. Точка.

Последние два — стохастические: генерируется случайный ключ, подписываются данные, проверяется подпись. Они не проверяют конкретный вектор — они проверяют, что цепочка sign→verify работает end-to-end. Если RSA sign возвращает мусор — verify это поймает.

`CryptoSelfTestError` при любом провале — и процесс завершается. Это требование FIPS 140-2, Section 4.9.1: «The module shall perform the self-tests at power-up and shall not perform any cryptographic function until the tests are completed successfully.» Мы не сертифицированы по FIPS, но следуем духу стандарта.

### TRNG — `core/trng.py` — источник непредсказуемости

Два класса, одна задача: дать системе настоящие случайные числа.

`SoftwareTRNG` — fallback. Смешивает `os.urandom(n)`, timestamp (`time.time() * 1e9`) и PID (`os.getpid() + os.getppid()`) через SHA-512. Три источника, каждый по отдельности предсказуем, но их комбинация через криптографический хеш — нет (при условии, что хотя бы один источник содержит энтропию). `os.urandom` на Linux читает `/dev/urandom` — это CSPRNG ядра, достаточно хороший для разработки, но не для production PKI.

`HardwareTRNG` — основной класс. Три режима:
- `hardware` — только USB HID. Если STM32 недоступен — `TRNGDeviceError`, система не стартует.
- `software` — только SoftwareTRNG. Для тестирования и CI/CD.
- `auto` — пробует hardware, при недоступности молча переключается на software.

`auto` — режим по умолчанию, и это осознанный компромисс. В production конфиг явно указывает `mode: hardware`. В development — `auto` позволяет работать без STM32. Разница — в одной строке YAML, но за ней — разница между «аппаратная энтропия гарантирована» и «аппаратная энтропия желательна».

Health check — два статистических теста на 2048 байтах:
- Bit ratio: доля единичных бит должна быть между 0.40 и 0.60. Идеал — 0.50. Отклонение больше 0.10 — bias в генераторе.
- Chi-square: распределение байтов должно быть равномерным. Chi-square > 310 (при 255 степенях свободы) — p-value < 0.01, то есть менее 1% вероятности, что данные случайны.

Эти тесты не гарантируют криптографическое качество — для этого нужен полный NIST SP 800-90B suite. Но они ловят грубые проблемы: залипший бит, сломанный генератор, подменённое устройство с PRNG.

### NIST DRBG — `core/drbg.py` — усилитель энтропии

80 строк кода, которые превращают 32 байта аппаратной энтропии в бесконечный поток криптографически стойких псевдослучайных чисел.

HMAC_DRBG по NIST SP 800-90A — один из трёх стандартных DRBG (наряду с Hash_DRBG и CTR_DRBG). Выбран HMAC потому что он проще в реализации (нет block cipher mode, нет padding) и потому что HMAC-SHA256 уже используется в системе (KeyStorage, self tests).

Три операции, три фазы жизни:
- `instantiate(personalization)` — первичный seed. 32 байта энтропии от TRNG + опциональная personalization string. Результат — инициализированное состояние (key + value).
- `generate(n, additional)` — генерация n байт. HMAC-SHA256 в цикле, каждая итерация обновляет value. После генерации — update состояния.
- `reseed(additional)` — обновление состояния свежей энтропией. Автоматически при `reseed_counter >= reseed_interval`.

`reseed_interval = 1000` — каждые 1000 вызовов `generate()` DRBG запрашивает свежие 32 байта от TRNG. Это не произвольное число — NIST SP 800-90A рекомендует reseed interval не более 2^48 для HMAC_DRBG, но мы используем 1000 для дополнительной безопасности. На практике это означает: при выпуске 1000 сертификатов подряд — один reseed, ~4ms задержки на USB HID read.

Реализация намеренно минимальна. Никаких оптимизаций, никаких нестандартных расширений, никакого кэширования. 80 строк, которые можно прочитать за 10 минут и сверить с NIST SP 800-90A Section 10.1.2. Для криптографического кода простота — это не лень, это аудируемость.

### CryptoEngine — `core/crypto_engine.py` — фасад над OpenSSL

CryptoEngine не реализует криптографию — он оркестрирует её. Его задача: убедиться, что OpenSSL использует аппаратную энтропию, и предоставить удобный интерфейс для остальной системы.

Главный метод — `_seed_openssl(n=64)`. Перед каждой генерацией ключей: взять 64 байта из DRBG, вызвать `RAND_add(data, len, float(len))`. Это подмешивает аппаратную энтропию в OpenSSL RAND pool. Без этого шага OpenSSL использовал бы только `/dev/urandom` — программный CSPRNG ядра, который на embedded-платформе с малым количеством источников энтропии может быть предсказуемым в первые минуты после загрузки.

Два семейства ключей:
- RSA: по умолчанию 4096 бит (из конфига `crypto.rsa_key_size`). `generate_rsa_keypair()` принимает override — CA использует 4096, серверные сертификаты — 2048.
- EC: по умолчанию P-384 (SECP384R1). Единственная поддерживаемая кривая — осознанное ограничение. P-256 слишком короткая для 20-летнего горизонта, P-521 — overkill и медленнее.

`sign_data()` — универсальный метод подписи. Определяет тип ключа (RSA или EC) и выбирает алгоритм: RSA-PKCS1v15-SHA256 или ECDSA-SHA256. Audit log записывает каждую операцию: алгоритм, размер данных. Это не для отладки — это для compliance: кто, когда, что подписал.

`verify_certificate()` — проверка подписи сертификата. Используется в церемонии (verify после sign) и может использоваться для валидации цепочки. Возвращает bool, не бросает исключение — потому что невалидный сертификат — это не ошибка программы, это факт, который нужно обработать.

### KeyStorage — `core/key_storage.py` — хранитель секретов

Самый параноидальный класс в системе. Его задача — убедиться, что приватные ключи никогда не существуют в открытом виде дольше, чем необходимо.

Шифрование: AES-256-GCM. Ключ шифрования выводится из пароля через PBKDF2-HMAC-SHA256 с 260 000 итерациями и 32-байтным случайным salt. 260 000 итераций — это ~0.5 секунды на RK3328. Для легитимного пользователя — незаметно (ключ загружается раз при старте). Для атакующего, перебирающего пароли — 260 000 хешей на каждую попытку.

Формат файла `.enc`: `[4 байта длины salt][salt][12 байт nonce][ciphertext+tag]`. Намеренно простой — никакого ASN.1, никакого PKCS#12, никаких вложенных структур. Чем проще формат — тем меньше парсера — тем меньше поверхность атаки.

GCM tag (16 байт) в конце ciphertext обеспечивает целостность. Если файл повреждён (битый диск, неполная запись) — расшифровка не вернёт мусор, а выбросит `InvalidTag`. Это важно: мусорный ключ хуже отсутствующего — он может подписать невалидный сертификат.

`_zeroize()` — два подхода для двух типов Python:
- `bytearray` — мутабельный, затирается поэлементно в цикле.
- `bytes` — иммутабельный в Python, но `ctypes.memset` обращается напрямую к буферу объекта в памяти, минуя Python runtime.

Это не гарантия (Python GC может создать копии), но это best effort. В C это было бы `explicit_bzero()`. В Python — `ctypes.memset` + надежда, что GC не успел скопировать.

`delete_key()` — тройная защита: перезапись нулями → `fsync` (нули дошли до диска, а не остались в кэше FS) → `unlink`. На SSD с wear leveling это не гарантирует физическое уничтожение (старые блоки могут остаться), но на eMMC RK3328 — достаточно.

## Storage Layer: два хранилища, две правды

Данные в PKI-on-Box хранятся дважды — и это не ошибка проектирования, а осознанное решение. SQLite хранит метаданные для поиска. Файловая система хранит артефакты для экспорта. Два хранилища, два формата, одна истина.

### PKIDatabase — `storage/database.py` — реестр

SQLite, один файл (`pki.db`), три таблицы. Выбор SQLite — не компромисс, а идеальное совпадение требований и возможностей.

Почему не PostgreSQL? Потому что на RK3328 с 2GB RAM PostgreSQL — это 200MB RAM на idle, WAL-файлы, vacuum, pg_stat. Для базы с тремя таблицами и тысячами записей — расточительство. SQLite — это один файл, zero configuration, zero maintenance. Он переживёт внезапное отключение питания (journal mode), не требует отдельного процесса и работает быстрее PostgreSQL для single-writer workload.

Три таблицы — три сущности:

`ca_certificates` — реестр CA. Поля: `id` (TEXT, primary key — `ca_root_ca`, `ca_intermediate_1`), `name` (человекочитаемое), `cert_pem` (полный PEM), `created_at` (ISO timestamp). Записей мало (2-5), но каждая критична.

`certificates` — реестр конечных сертификатов. Поля: `serial` (TEXT hex, primary key), `common_name`, `cert_pem`, `ca_id` (FK к ca_certificates), `issued_at`, `expires_at`, `status` (active/revoked). Serial как TEXT, не INTEGER — потому что X.509 serial number может быть до 20 байт (160 бит), что не влезает в INT64.

`revoked_certificates` — реестр отзывов. Поля: `serial` (TEXT hex, primary key), `reason` (enum name: keyCompromise, superseded, ...), `revoked_at`. Отдельная таблица, не флаг в `certificates` — потому что CRL генерируется из этой таблицы напрямую (`get_revoked(ca_id)` с JOIN), и отдельная таблица делает этот запрос проще и быстрее.

`_init_schema()` использует `CREATE TABLE IF NOT EXISTS` — идемпотентная инициализация. Первый запуск создаёт таблицы, последующие — пропускают. Никаких миграций, никакого Alembic — схема фиксирована и не меняется.

### CertificateFileStorage — `storage/file_storage.py` — экспорт

Файловая система — это интерфейс к внешнему миру. Когда клиенту нужен сертификат в PEM — он лежит в файле, готовый к `cat` или `scp`. Когда нужен DER — `export_der()` конвертирует на лету.

Naming convention: `{serial_hex}.pem`. Serial `0xa3f7b2c1d4e5` → файл `a3f7b2c1d4e5.pem`. Предсказуемо, уникально, но нечитаемо для человека.

Поэтому `by_label/` — индекс с человекочитаемыми именами. `store_cert(cert, label="server.example.com")` создаёт два файла: `a3f7b2c1d4e5.pem` (каноническое имя) и `by_label/server.example.com.pem` (копия). Не символическая ссылка, а полная копия — потому что на некоторых embedded-файловых системах (JFFS2, UBIFS) symlinks работают нестабильно.

`list_certs()` — glob `*.pem` в base directory, парсинг каждого PEM, извлечение serial и CN. Медленно для тысяч сертификатов, но для типичного deployment (десятки-сотни) — приемлемо. Для поиска по метаданным — SQLite.

## Security Layer (BSW): защита, которая не знает о Python

Этот слой принципиально отличается от всех остальных. Он написан не на Python, а на C (eBPF) и декларативных языках (SELinux TE, systemd unit files). Он работает не в userspace, а в ядре. И он защищает систему от ситуации, когда весь Python-код уже скомпрометирован.

Три механизма, три уровня, ноль зависимостей от приложения.

### SELinux Policy — `bsw/selinux/` — кто что может

Mandatory Access Control — это система, где права доступа определяются не владельцем файла (как в DAC), а политикой, которую нельзя изменить из userspace. Даже root не может обойти SELinux в enforcing mode (без `setenforce 0`, который требует CAP_SYS_ADMIN).

Два домена — два мира:

`pki_core_t` — основной сервис. Может: открывать TCP-сокеты (REST API), читать/писать `pki_var_t` (данные), дописывать `pki_log_t` (логи), читать `pki_config_t` (конфиг), подключаться к `pki_hsm_t` через unix_stream_socket. Не может: читать `/dev/hidraw0`, модифицировать свой бинарь, открывать произвольные файлы.

`pki_hsm_t` — HSM/TRNG мост. Может: читать/писать `/dev/hidraw0` (USB HID), читать/писать `pki_var_t`, дописывать `pki_log_t`. Не может: открывать TCP-сокеты, подключаться к сети. Capability `chown` и `dac_override` — для управления правами на файлы ключей.

Пять типов файлов — пять уровней доступа:

| Тип | Путь | Кто читает | Кто пишет |
|-----|------|-----------|----------|
| pki_core_exec_t | /usr/bin/pki-core | systemd (exec) | никто (immutable) |
| pki_hsm_exec_t | /usr/bin/pki-hsm | systemd (exec) | никто (immutable) |
| pki_var_t | /var/lib/pki-box/ | core, hsm | core, hsm |
| pki_log_t | /var/log/pki-box/ | core, hsm | core, hsm (append only) |
| pki_config_t | /etc/pki-box/ | core, hsm | никто (admin only) |

Ключевое ограничение: `pki_core_t` не может обращаться к USB HID. Даже если атакующий получит RCE через уязвимость в Flask и окажется в контексте `pki_core_t` — он не сможет напрямую читать аппаратный TRNG. Для доступа к энтропии ему придётся пройти через HSM Service (unix_stream_socket), который работает в отдельном домене со своими ограничениями.

Три файла политики:
- `pki-box.te` — Type Enforcement: правила доступа (allow, dontaudit, type_transition)
- `pki-box.fc` — File Contexts: какой тип у какого файла
- `pki-box.if` — Interfaces: макросы для других модулей (pki_domtrans, pki_read_config, pki_manage_storage)

### eBPF — `bsw/ebpf/` — наблюдение на уровне ядра

Два eBPF программы, две задачи: фильтрация сети и аудит системных вызовов.

`network_filter.c` — socket filter. BPF_MAP_TYPE_ARRAY с whitelist портов (максимум 16). Логика: если TCP пакет идёт на порт из whitelist — пропустить. Всё остальное — drop. Проще и надёжнее iptables: BPF-программа компилируется в байткод, верифицируется ядром (BPF verifier гарантирует, что программа завершится и не обратится к невалидной памяти), и выполняется в sandbox. Её нельзя модифицировать из userspace после загрузки — только выгрузить и загрузить заново (что требует CAP_BPF).

`syscall_filter.c` — tracepoint на `raw_syscalls/sys_enter`. Не блокирует — наблюдает. Три BPF_MAP:
- `pki_monitored_pids` (HASH, 16 entries) — какие PID мониторить
- `syscall_counter` (HASH, 512 entries) — счётчик вызовов по PID+syscall_nr
- `syscall_events` (PERF_EVENT_ARRAY) — real-time output: {pid, syscall_nr, comm[16]}

Почему аудит, а не enforcement? Потому что ложные срабатывания в enforcement могут уронить production. Если PKI Core вызывает `mmap` с `PROT_EXEC` (что `MemoryDenyWriteExecute` в systemd уже блокирует) — eBPF это запишет в лог, но не убьёт процесс. Два уровня защиты: systemd блокирует, eBPF наблюдает.

### systemd Sandboxing — последний рубеж

systemd unit files — это не просто «как запустить процесс». Это декларативная политика безопасности, которая применяется ядром при создании процесса.

| Директива | Что делает | Почему важно |
|-----------|-----------|-------------|
| ProtectSystem=strict | Root FS read-only | Атакующий не может модифицировать системные файлы |
| ProtectHome=yes | /home недоступен | Нет доступа к пользовательским данным |
| PrivateTmp=yes | Изолированный /tmp | Нет cross-process утечек через /tmp |
| NoNewPrivileges=yes | Запрет setuid/setgid | Эскалация привилегий невозможна |
| MemoryDenyWriteExecute=yes | Нет W+X памяти | Убивает большинство shellcode-техник |
| LockPersonality=yes | Запрет personality() | Нет переключения ABI (защита от некоторых exploit-техник) |
| RestrictRealtime=yes | Запрет RT scheduling | Нет DoS через RT-приоритет |
| RestrictSUIDSGID=yes | Запрет создания SUID файлов | Нет persistence через SUID |

Для HSM Service — дополнительно:
- `DeviceAllow=/dev/hidraw0 rw` — единственное устройство, к которому есть доступ
- `RestrictAddressFamilies=AF_UNIX` — только Unix-сокеты, TCP невозможен
- `RestrictNamespaces=yes` — нет создания namespace (защита от container escape)

Четыре механизма безопасности (SELinux + eBPF network + eBPF syscall + systemd) работают независимо. Если один отключён или обойдён — остальные три продолжают защищать. Это не redundancy — это defense in depth.

## Firmware Layer: 30KB кода, который стоит за всей криптографией

Firmware — самый маленький слой по объёму кода и самый важный по влиянию. Если firmware генерирует предсказуемые числа — вся PKI бессмысленна. Никакой DRBG, никакой OpenSSL, никакой SELinux не компенсирует слабый источник энтропии.

### Архитектура firmware — `firmware/hmi/`

Bare metal, без ОС. PlatformIO + STM32Cube HAL. Три файла, которые имеют значение:

`main.c` — точка входа и main loop. Последовательность инициализации жёсткая:

```
HAL_Init() -> SystemClock_Config() -> LED_Init()
  -> TRNG_Init() -> TRNG_StartupTest(TSR-1)
  -> IWDG_Init() -> USB_Init()
  -> main loop: IWDG_Refresh -> TRNG_FillReport -> USB_SendReport
```

Каждый шаг может завершиться ошибкой, и каждая ошибка — fatal. `Error_Handler_Ex(code)` отключает прерывания и входит в бесконечный цикл SOS-мигания светодиодом. Никаких retry, никаких fallback. Если RNG peripheral не инициализировался — MCU мигает SOS до перезагрузки. Это не баг — это единственное честное поведение для генератора случайных чисел: если не можешь генерировать — не притворяйся.

`trng_hid.c` — ядро: инициализация RNG, health tests, заполнение USB HID репортов.

`TRNG_Init()` настраивает тактирование RNG peripheral. На G4 серии — HSI48 (внутренний 48 МГц RC-осциллятор) или PLL. На H7 — другая схема тактирования, отдельный `#ifdef`. Если `HAL_RNG_Init()` возвращает ошибку — `ERR_RNG_INIT`, SOS.

`TRNG_StartupTest()` (TSR-1) — тест при включении. Генерирует два 32-битных слова и проверяет:
- Stuck-at: ни одно слово не должно быть 0x00000000 или 0xFFFFFFFF. Если генератор залип на нулях или единицах — `ERR_TSR1_STUCK`.
- Equality: два слова не должны быть равны. Вероятность случайного совпадения — 1/2^32. Если совпали — `ERR_TSR1_EQUAL`.

`TRNG_FillReport()` — заполнение 64-байтного репорта в main loop. 16 вызовов `HAL_RNG_GenerateRandomNumber()`, каждый даёт 4 байта. Между вызовами — проверка флагов:
- `SECS` (Seed Error) — seed для внутреннего кондиционера не прошёл проверку → `ERR_RNG_SECS`
- `CECS` (Clock Error) — тактовый генератор RNG нестабилен → `ERR_RNG_CECS`

И continuous test (TSR-2):
- Repetition: текущее слово == предыдущее → `ERR_TSR2_REP`
- Proportion: MSB четырёх последовательных слов одинаков → `ERR_TSR2_PROP`

Коды ошибок — не просто числа, а диагностика:

| Код | Hex | Что сломалось | Что делать |
|-----|-----|--------------|-----------|
| ERR_RCC_CLK | 0x10 | Тактирование | Проверить кварц/PLL |
| ERR_RNG_INIT | 0x11 | RNG peripheral | Проверить HAL config |
| ERR_RNG_GEN | 0x12 | Генерация числа | Аппаратный дефект |
| ERR_RNG_SECS | 0x13 | Seed error | Аппаратный дефект |
| ERR_RNG_CECS | 0x14 | Clock error | Проверить тактирование |
| ERR_TSR1_STUCK | 0x20 | Залипание при старте | Аппаратный дефект |
| ERR_TSR1_EQUAL | 0x21 | Повтор при старте | Аппаратный дефект |
| ERR_TSR2_REP | 0x30 | Повтор в runtime | Деградация генератора |
| ERR_TSR2_PROP | 0x31 | Смещение в runtime | Деградация генератора |

Watchdog (IWDG): prescaler=64, reload=1000 → ~4 секунды. Если main loop зависнет (бесконечный цикл в USB stack, deadlock в HAL) — MCU перезагрузится через 4 секунды. После перезагрузки — снова startup test, снова инициализация. Автоматическое восстановление без участия хоста.

LED: 100ms blink когда USB connected, 500ms когда idle. Визуальная диагностика без осциллографа — если светодиод мигает быстро, firmware работает и USB активен.

## Эпилог: 16 компонентов, одна цепочка

Если посмотреть на все 16 компонентов сверху, видна одна структура — цепочка, не дерево. Каждый компонент получает доверие от предыдущего и передаёт следующему:

```
Тепловой шум (физика)
  -> STM32 RNG peripheral (аппаратура)
    -> Firmware health tests (bare metal)
      -> USB HID transport (протокол)
        -> HardwareTRNG (Python, host)
          -> NIST DRBG (криптографическое расширение)
            -> CryptoEngine (OpenSSL seeding)
              -> KeyStorage (шифрование ключей)
              -> CA Service (управление CA)
                -> Certificate Service (выпуск)
                -> CRL Service (отзыв)
                -> OCSP Responder (статус)
              -> PKIDatabase + FileStorage (персистентность)
            -> REST API + CLI (интерфейс)
          -> SecurityManager (изоляция)
```

Разорви любое звено — и всё, что выше, теряет смысл. Убери TRNG — DRBG получит предсказуемый seed. Убери KAT — сломанный OpenSSL будет генерировать слабые ключи. Убери SELinux — RCE в Flask даст доступ ко всем ключам.

16 компонентов, 12 Python-файлов, 2 C-файла, 3 systemd unit, 3 SELinux policy файла. ~4000 строк кода, которые превращают тепловой шум в кремнии STM32 в X.509 сертификаты, подписанные RSA-4096, защищённые AES-256-GCM, изолированные SELinux и eBPF.

Стоимость: $129 и массив хороших инженерных решений.
